# Nightjar Feasibility Audit
Offline local-first AI coding + personal assistant — **licensed AGPL-3.0-or-later**, open-source, personal GitHub (relicensing decision confirmed; supersedes the earlier "closed BlackKrait product" framing that shaped some prior UI-license analysis).
Prepared from clones in `research/` (opencode, row-bot, odysseus, open-webui, gemma-chat, orb-ui — github.com/alexanderqchen/orb-ui). LobeChat was evaluated first and replaced by Open WebUI as the UI candidate/reference (see §5, §4). **Odysseus (AGPL-3.0-or-later) added as a third capability tier — see `ODYSSEUS_INTEGRATION_PLAN.md` for the full findings + integration plan.**

**License direction (AGPL-3.0-or-later combined work):** the AGPL decision is clean — OpenCode (MIT) and Row-Bot (Apache-2.0) are one-way compatible into AGPL, and Odysseus is the AGPL anchor. Bridging over MCP (separate processes) is compatible under both the strict "single combined work" and lenient "mere aggregation" readings. Preserve all upstream attribution: OpenCode MIT notice, Row-Bot NOTICE, Odysseus LICENSE + `licenses/` + ACKNOWLEDGMENTS.md. Watch-item: AGPL §13 network clause if any hosted/remote-access mode is ever added; PyMuPDF (AGPL, optional) and SearXNG (AGPL, composed service) if used. The earlier closed-product UI-license concerns (LobeChat commercial-derivative, Open WebUI branding-lock) are now moot for a fork *decision* but both are still ruled out on framework/scope grounds — UI stays custom React.

OpenCode and Row-Bot are **different runtimes entirely** (Bun/TypeScript/Effect vs. Python/LangGraph) — there is no native compatibility. The correct integration shape is: **OpenCode is the base engine and the only agent loop; Row-Bot is decapitated of its own orchestrator and re-exposed as a set of tools OpenCode calls via MCP.**

Nightjar's UI will be built in **React** regardless of which reference project was considered (LobeChat, then Open WebUI) — both were ruled out for forking, and orb-ui's hard React dependency locks the UI framework choice in anyway (see §5, §6).

## 1. OpenCode — Architecture

- **Runtime**: Bun + TypeScript, Turborepo monorepo, 34 workspace packages, built on the **Effect** functional-effects library and Vercel's **AI SDK** (`ai` package).
- **Note on provenance**: root `package.json` and docs reference `github.com/anomalyco/opencode`, not `sst/opencode` — the project appears to have moved orgs. Worth re-verifying you're tracking the canonical upstream before building on it.
- **Agent loop**: `packages/opencode/src/session/processor.ts` — a session is a durable SQLite-backed entity (Drizzle ORM). Each turn streams from the model via `packages/opencode/src/session/llm.ts`, which calls the AI SDK's `streamText()`; every message "part" (text, tool-call, reasoning, patch) is persisted incrementally so state survives crashes/restarts. The loop is classic ReAct: keep calling the model while it requests tools, stop when it doesn't, auto-compact context on overflow. Includes a "doom loop" breaker (3 identical tool calls → permission prompt).
- **Provider abstraction**: `packages/opencode/src/provider/provider.ts` — a registry of ~20 named providers plus a generic `@ai-sdk/openai-compatible` fallback for anything unlisted. Model catalog sourced from the external Models.dev registry, merged with user/plugin config.
- **Local-model support**: real but generic — **no bespoke Ollama module**. Ollama (and llama.cpp, LM Studio, vLLM, etc.) are configured as a plain OpenAI-compatible endpoint (`baseURL: http://localhost:11434/v1`) via the same adapter used for cloud OpenAI-compatible providers. Docs flag real friction: tool-calling reliability with local models depends on the local server's function-calling emulation, and `num_ctx` often needs manual tuning.
- **Extension points** (this is the important part for grafting Row-Bot on):
  1. **Plugin system** (`@opencode-ai/plugin`, MIT) — a plugin exports hooks: `tool` (register new tools with Zod-schema args and an `execute()` function), `auth`, `provider`, `chat.message`/`chat.params`, `permission.ask`, `tool.execute.before/after`. Loaded from `.opencode/plugins/` or npm packages listed in config. This is a ~50-line API surface — adding a capability (voice, vision, browser) is a new tool file, not a core fork.
  2. **MCP (Model Context Protocol) client** (`packages/opencode/src/mcp/`) — OpenCode can attach to any MCP server declared in `opencode.json`, with no OpenCode-specific code needed on the server side.

## 2. Row-Bot — Architecture

- **Runtime**: Python ≥3.12, `uv` package manager, LangChain + LangGraph for orchestration, NiceGUI/FastAPI for its own web UI.
- **Core loop**: `src/row_bot/agent.py` (~4900 lines) builds a LangGraph `create_react_agent` with a `SqliteSaver` checkpointer and `interrupt()`-based approval gates for destructive tools. All input — typed chat and voice-transcribed text — funnels through the same `stream_agent()` call. This is a **second, independent agent loop** with its own LLM-calling, state, and interruption semantics, conceptually redundant with OpenCode's `processor.ts` if both were run.
- **Voice**: default path is fully local and offline — `faster-whisper` (STT, CPU int8) and `kokoro-onnx` (TTS, local ONNX model, explicitly documented as "no audio sent to the cloud"). An optional cloud path (OpenAI Realtime API) also exists, opt-in only.
- **Vision**: `src/row_bot/vision.py` captures camera (OpenCV) or screenshots (mss), then branches to either a cloud multimodal LLM call or a **local Ollama vision model** (default `gemma3:4b`, also supports moondream/llava family) — fully local-capable.
- **Memory**: hybrid system in `src/row_bot/knowledge_graph.py` — SQLite as source of truth, a NetworkX graph mirror, and a FAISS semantic index, with retrieval combining vector search + FTS5/BM25 keyword search + 1-hop graph expansion. Runs fully local with local embeddings (`HuggingFaceEmbeddings`) or optionally cloud embeddings.
- **Browser automation**: `src/row_bot/tools/browser_tool.py`, Playwright (sync API) with a persistent Chromium profile and per-thread tab isolation — a real, non-trivial implementation (1370 lines), not a toy wrapper.
- **Coupling assessment (critical finding)**: the four subsystems — voice, vision, memory, browser — are **loosely coupled, near-zero-LangChain plain Python modules**. LangChain only appears in thin `as_langchain_tools()` adapter methods (tens of lines each) that wrap each module as a `StructuredTool`. What **is** tightly bound to LangGraph and not portable: the `create_react_agent` graph construction, the `SqliteSaver` checkpointer, and the `interrupt()`/`Command` resume flow for approvals — i.e., Row-Bot's *orchestration shell*, not its capabilities.

## 3. Integration Complexity: OpenCode ↔ Row-Bot

**They are not compatible stacks.** OpenCode is Bun/TypeScript/Effect; Row-Bot is Python/LangGraph. There is no way to "merge" them at the process/import level — a bridge is required regardless of approach.

**Which is the base, which is the bolt-on:** OpenCode should be the base. Reasons:
- It already has the more general, better-abstracted agent loop (provider-agnostic, plugin/MCP extensible, durable SQLite session state, doom-loop protection, context compaction).
- Running *both* agent loops (OpenCode's and Row-Bot's LangGraph loop) simultaneously would mean two independent LLM-calling orchestrators fighting over turn-taking — architecturally wrong, not just inconvenient.
- Row-Bot's genuinely valuable, hard-to-rebuild parts (local voice pipeline, local vision routing, the hybrid graph+vector memory engine, the Playwright browser tool with persistent-profile/tab isolation) are, per the coupling assessment above, already close to standalone.

**Recommended bridge: MCP, not a language-level port.**
- Discard Row-Bot's `agent.py` (LangGraph graph, checkpointer, approval-interrupt flow), its NiceGUI/FastAPI app, and its `ChatOllama`/provider-selection layer entirely — none of that should run in Nightjar; OpenCode replaces all of it.
- Extract `voice/`, `vision.py`, `knowledge_graph.py`/`memory.py`, and `tools/browser_tool.py` (plus their direct, non-LangChain dependencies) into a standalone Python package/process.
- Wrap that package as an **MCP server** (Python has mature MCP server SDKs; Row-Bot even already has its own `mcp_client/` module for the reverse direction, confirming the team is MCP-literate). Expose `transcribe`, `speak`, `analyze_image`, `capture_screen`, `save_memory`/`search_memory`, and `browser_navigate`/`browser_click`/etc. as MCP tools.
- Point OpenCode at this MCP server via `opencode.json` — zero OpenCode core changes required; this is exactly the extension point `packages/opencode/src/mcp/` exists for.
- This avoids reimplementing faster-whisper/kokoro-onnx/FAISS/Playwright glue in TypeScript (real, non-trivial work) while avoiding two competing agent loops.

**Realistic complexity**: moderate, not small. Non-trivial parts: (1) turning long-running stateful things (a persistent browser session, a live microphone stream) into clean MCP tool calls, since MCP's request/response tool model doesn't map perfectly onto "start listening and stream partial transcripts" — voice will likely need either a push-based MCP notification pattern or a separate lightweight local WebSocket/IPC channel alongside MCP; (2) the memory system's auto-recall-into-prompt behavior (`memory_policy.py`) is currently done inside Row-Bot's own LangGraph turn — that logic (deciding *when* to inject recalled memories) will need to move into an OpenCode plugin hook (`chat.message`/`chat.params`) rather than the MCP server itself, since MCP servers don't see the full prompt-construction pipeline.

## 5. Open WebUI / gemma-chat: Fork vs. Reference

**Open WebUI: use as visual/UX reference only, do not fork.** Two independent, each-sufficient reasons — one licensing, one architectural:

1. **License** — see §4. The branding-lock clause is a material condition of the license grant, gated at 50 users, backed by a live license-verification service. Any real Nightjar deployment would need either a purchased enterprise license or to live with "Open WebUI" branding baked into a product sold under BlackKrait's name — neither is acceptable for "ship this as Nightjar."
2. **Frontend framework mismatch — decisive on its own, independent of the license.** Open WebUI's frontend is confirmed **SvelteKit 2 + Svelte 5** (593 `.svelte` files, zero `.tsx`/`.jsx`/`.vue` anywhere, `svelte`/`@sveltejs/kit` in `package.json`, zero `react`/`vue` dependencies) — not React. Since orb-ui (§6) is a React-only component with a hard `react >=18` peer dependency and no Web Component/vanilla build, forking Open WebUI's Svelte shell would mean either running two UI frameworks side-by-side in one bundle (Svelte + React + ReactDOM, two reconciliation loops, manual state bridging) or replacing the orb visual entirely. Neither is a real integration path — it's a rewrite in disguise.

Also worth weighing even if both blockers were removed: Open WebUI is large and mature (~102k LOC backend, ~138k LOC frontend, 30 backend routers, 13 vector-DB backends, 25+ web-search integrations, SCIM/enterprise auth, calendar/automations/channels) — a full platform built around a hosted multi-tenant Ollama-proxy product, not a lean single-user offline assistant shell. Stripping it down would likely cost more than building custom, on top of the license and framework problems.

Worth studying (not copying) from Open WebUI: its **Ollama multi-instance proxy/load-balancing design** (`backend/open_webui/routers/ollama.py` — random load-balanced dispatch across multiple configured Ollama base URLs, with per-request backend pinning) is directly relevant if Nightjar ever needs to talk to more than one local model server; its RAG/vector-DB abstraction (`retrieval/vector/factory.py`, a clean `VectorDBBase` interface behind 13 swappable backends) is a good reference shape for a pluggable memory backend; and its chat streaming design (Socket.IO event bus with typed `chat:message:delta` events, `Chat.svelte`) is a reasonable pattern to emulate for OpenCode's own SSE/tool-call event stream.

**gemma-chat: reference only, not a foundation.** License is a non-issue (MIT), but it's a ~4,400-line single-purpose proof-of-concept: no state management, no plugin system, no persistence beyond flat files, hard-locked to Apple-Silicon MLX-LM and the Gemma model family. The one genuinely reusable idea is the **live-preview coding pattern**: the agent streams XML-tagged file-write actions mid-generation (chosen over JSON tool-calls because small local models emit XML more reliably) into a sandboxed per-conversation workspace, which is served by a local static HTTP server and shown live in an iframe (`Canvas.tsx`), with a synchronized "streaming file" code view. This maps directly onto what a coding assistant needs and is worth re-implementing (not literally copying) against OpenCode's own tool-call/patch events. gemma-chat is also already React (Electron + React 19), consistent with the framework Nightjar is committed to.

**Verdict: build Nightjar's UI from scratch in React**, using Open WebUI and gemma-chat purely as design/UX references — Open WebUI for chat/streaming/RAG UI conventions and Ollama-proxy design, gemma-chat for the live-preview canvas mechanism.

## 6. orb-ui — Architecture and Integration Path

**Framework**: React (TypeScript) component library, `peerDependencies: react >=18`. Published to npm as `orb-ui` v0.2.4 (ESM+CJS, real `exports` map, Changesets/CI release pipeline) — not a copy-paste snippet, a genuine dependency you'd `npm install`. Requires React; not usable as a bare `<script>` include or in a non-React UI shell without a React runtime embedded.

**Rendering**: plain DOM `<span>` elements with inline CSS (`transform`/`boxShadow`/`background`) animated via `requestAnimationFrame` — **not** Canvas/WebGL/shader-based. Three built-in themes: `circle`, `bars`, `debug` (a diagnostic readout, not for production). Lightweight and fine performance-wise for a single orb, but visually simpler than a shader-driven "orb" effect; colors and volume-sensitivity curves are hardcoded per theme file with no `color`/`palette` prop — reskinning means forking a theme file (`CircleTheme.tsx`/`BarsTheme.tsx`, ~300 lines each).

**Component API — this is the key integration fact**: the component's entire audio-input surface is a single normalized `volume: number` (0–1) prop plus a `state: 'idle'|'connecting'|'listening'|'speaking'|'error'` prop, used in "controlled mode":
```tsx
<Orb state={state} volume={volume} theme="circle" />
```
There is no prop accepting a raw `MediaStream`, `AnalyserNode`, or frequency-bin array — you must reduce your own audio to one scalar before passing it in. (There's also an "adapter" mode built for third-party voice SDKs — Vapi, ElevenLabs — that includes a private, undocumented mic-monitor that monkey-patches `navigator.mediaDevices.getUserMedia` globally; this code path never runs in controlled mode, which is what Nightjar should use, so it's a non-issue as long as you don't pass an `adapter` prop.)

**Wiring it to Row-Bot's local voice pipeline** (Phase 2's `faster-whisper`/`kokoro-onnx` MCP server): there's no built-in mic-input example in the repo (its own demo only uses a scripted sine-wave simulation), so this is custom glue either way — standard Web Audio API pattern: create an `AnalyserNode` on the mic stream (listening) or on the Kokoro-synthesized audio output (speaking), call `getByteFrequencyData` each frame, reduce to an RMS scalar (orb-ui's own private reduction in `Orb.tsx` — sum-of-squares → sqrt(mean)/255 with EMA smoothing — is a reusable reference pattern even though it's not exported), normalize to 0–1, and feed it into React state driving the `volume` prop. Drive `state` from the voice pipeline's own lifecycle (listening while recording, speaking while Kokoro plays back). The repo's own convention for formalizing this is a custom `OrbAdapter` object (`subscribe({onStateChange, onVolumeChange})`), the same shape as its existing Vapi/ElevenLabs adapters — worth mirroring for a "local pipeline adapter."

**Maturity**: small (1,672 LOC), single-maintainer, pre-1.0 (v0.2.4, roadmap explicitly plans a breaking "richer adapter signal model"), thin test coverage (accessibility/DOM-shape assertions only, no tests on the volume/animation math). Legitimately shippable and easy to fully audit or fork given its size, but budget for API churn before 1.0 and expect to fork the theme files for custom branding/colors.

**Verdict**: use directly (bolt-on, not reference-only) — MIT license, small enough to vendor if needed, and its controlled-mode API is a clean, if minimal, integration point for a custom local audio pipeline.

## 7. Phased Build Plan

**Phase 1 — Engine (base) — ✅ DONE.** Full findings in `phase1-engine/PHASE1_REPORT.md`; summary:
Bun + OpenCode (built from source — the published npm binary launcher expects a prebuilt platform binary that doesn't exist for a from-source clone; use `bun run --cwd packages/opencode --conditions=browser src/index.ts` instead) + local Ollama, configured via the generic `openai-compatible` adapter, produces a real working end-to-end loop: genuine `read`/`edit`/`bash`/`grep` tool calls against the filesystem, not simulated. **Neither originally-proposed model survived contact with reality**: Qwen3-Coder only ships at 30B+ on Ollama (too large for the 6GB-VRAM laptop GPU this was tested on) and Gemma3 4B is flatly rejected by Ollama's own API ("does not support tools") — confirming the tradeoff flagged earlier in this doc. The substitute, `qwen2.5-coder:7b-instruct`, also completely failed tool-calling (100% failure rate across native and OpenAI-compat transports, unfixable by `num_ctx` tuning — a model/template incompatibility, not an OpenCode bug). Landed on the **Qwen3 family** with a custom 32k-context Modelfile baked in (empirically required, not just documentation advice). The most important finding: **tool-call format reliability and tool-use judgment safety are separate axes** — `qwen3:1.7b` formats tool calls correctly but destructively overwrote a test file when its first edit attempt failed validation (fell back to a full-file `write` instead of fixing the edit args); `qwen3:8b` fixed this specific failure in direct side-by-side testing, but only 40% fits in 6GB VRAM at 32k context, making it 2-4x slower. This latency-vs-safety tradeoff is now a real product decision for Phase 2, since several of Row-Bot's planned MCP tools (browser automation, memory writes) are exactly the "destructive if judgment fails" category this phase found a live example of. Also flagged: headless `run` auto-rejects permission prompts without `--auto` (a real policy decision needed before Phase 2's daemon mode), and OpenCode's `grep`/`glob` tools download a `ripgrep` binary on first use — needs pre-bundling for a genuinely offline install.

**Phase 1.5 — Model swap + safety harness — ✅ DONE.** Full findings in `phase1-engine/PHASE1.5_REPORT.md`; summary:
Default model swapped to **Qwen3-4B-Instruct-2507 Q4_K_M served by llama.cpp** (built from source with CUDA for the RTX 4050) instead of Ollama — `--jinja` gives proper structured `tool_calls` (fixing the Phase 1 format problem at the server layer), the 4B model fits *fully* in 6GB VRAM (5.4GB resident), and edits complete in **~18s** vs ~13min for the 8B partial-offload config. Critically, the 4B model does **non-destructive edits natively** (proper `edit`, not the destructive full-file `write` that qwen3:1.7b used). A **safety harness of OpenCode plugins** was built and validated (14/14 deterministic unit tests + end-to-end): (A) `no-destructive-write` blocks a `write` that would shrink an existing file below 50% size / 50% line-preservation, steering the model back to `edit` — confirmed blocking a real destructive write e2e; (B) `git-gate` scope-checks each edit via `git status` and rolls back out-of-scope changes with checkpoint commits — *a real bug was found and fixed here: the first version treated pre-existing uncommitted files as out-of-scope and deleted them, a cautionary data point that over-broad auto-rollback is itself a data-loss risk*; (C) `doom-loop` hard-denies the built-in `doom_loop` permission (which `--auto` would otherwise auto-approve, defeating the loop breaker) plus an independent stricter counter. Hardware detection (`hw-detect.mjs` + a startup-logging plugin) reports RAM/VRAM and recommends a model tier keyed off *total* VRAM capability — detection + logging only, auto-switching deliberately deferred.

**Stuck-generation timeout — FIXED (closes the Phase 1.5 hazard before Phase 2).** The one open hazard (a single model call running unbounded — a ~300s repetition loop toward the context limit, which the doom-loop guard can't catch because it's one call, not repeated completed calls; root cause: llama.cpp `--predict` defaults to -1/infinity) is fixed with three complementary, validated layers: (1) a **wall-clock timeout proxy** (`inference-proxy.mjs`, Bun) between OpenCode and llama-server that hard-aborts any request past a deadline *mid-stream* — validated aborting a long generation at 3.013s against a 3s deadline, with the abort propagating to free the server slot (next request served in 0.375s); (2) a **token-cap plugin** (`chat.params` → `maxOutputTokens`, default 2048) that bounds a loop earlier/cheaper; (3) **server-side `--predict 2048` + `--timeout 120`** as a client-independent backstop. Token cap bounds length, proxy bounds time, server cap is independent — no single call can run unbounded. Runtime topology is now `OpenCode → :8086 proxy → :8085 llama-server`; the proxy also becomes the supervised inference endpoint Phase 2/3 needs. Plugin unit tests now 17/17.

**Pre-request freeze — FIXED with a run-supervisor watchdog (`nightjar-run.mjs`).** Distinct from unbounded generation: OpenCode can rarely hang in *session setup before sending any request* (observed only under near-full memory). Neither the proxy (nothing to abort) nor a plugin (process wedged) can catch it — it needs an external supervisor. The watchdog launches the run in its own process group and polls the proxy's new `/nightjar/stats` progress counter (incremented only on real generation requests); if no request reaches the model within a deadline (default 45s) and the run is still alive, it kills the frozen process group and **relaunches automatically** (up to 3 attempts), transparently — the user never sees the freeze or needs to retry. Exhausted attempts exit with a user-actionable "almost always low memory" message. Validated 9/9 (`verify-watchdog.sh`: happy path, freeze→auto-recover-on-attempt-2, persistent-freeze exhaustion, clean process-group kill) plus real-engine handoff. The three guards are now clean and non-overlapping — **watchdog**: run never reaches the model → restart; **proxy**: request reached the model but won't finish → abort; **doom-loop plugin**: repeated completed identical calls → block. Nightjar's engine driver will invoke runs through this watchdog in later phases.

**Phase 2 — Integration (Row-Bot capabilities as an MCP bolt-on) — ✅ DONE.** Full findings in `phase2-mcp/PHASE2_REPORT.md`; summary:
Voice/vision/memory/browser extracted into a standalone `nightjar_capabilities` package under `phase2-mcp/`, with LangChain/LangGraph/NiceGUI dropped — the four Row-Bot modules are vendored faithfully (Apache-2.0, NOTICE preserved) with orchestration/config glue replaced by small stubs and only 3 targeted code patches (browser headless, bundled Chromium, one dead langchain import). Everything runs **fully local and torch-free**: faster-whisper (CTranslate2), kokoro-onnx, openWakeWord (onnxruntime), embeddings swapped to Ollama `nomic-embed-text`, vision via Ollama `gemma3:4b`. Wrapped as a **14-tool MCP server** (stdio, FastMCP) registered in `opencode.json` — OpenCode reports `✓ nightjar connected`, no core changes. Auto-recall moved into an OpenCode `chat.message` **plugin** (validated: saved "user is Axe" → asked "what name do I go by?" → model answered "Axe" from injected memory). ### ARCHITECTURAL CONSTRAINT (required, not optional): per-mode agent scoping
A single flat tool list of ~53 tools (native OpenCode ~14 + Row-Bot `nightjar` 14 + Odysseus 25) is **empirically unreliable** for the local 4B model (Phase 2b probes: 2/4 correct; picked no tool for a memory save, and a colliding `manage_rag` instead of `document_search`). **The system MUST operate through scoped OpenCode agents** — each agent exposes only a small, task-relevant tool subset via its `tools` allow/deny map (`{"*": false, "<tool>": true, …}`, which compiles to permission rules; `findLast` precedence scopes to exactly the listed tools even while all servers stay registered). The "one giant tool list" mode is **not a supported operating mode**. **⚠️ Phase-3 correction: scoping and gating are DIFFERENT levers.** The `tools: {x:true}` map compiles to an *allow* rule — it makes a tool visible AND auto-approves it (no permission prompt). To keep a tool visible **and gated** (so `permission.asked` fires — needed for email-send/edit/bash approval), define it in the agent's **`permission` field** as `"ask"` (`permission:{x:"allow"|"ask"|"deny"}`); reserve the `tools` boolean map for hard-hiding (`x:false`). So mode definitions should use `permission` per-tool for anything that should prompt, not `tools:true`. Three agent modes exist today (in `phase2-odysseus/workspace/opencode.json`; to be updated to the `permission`-field form for gated tools):

| Agent mode | ~Tool count | Scoped tools |
|---|---|---|
| **research** | 2 | `deep_research`, `send_email` |
| **assistant** | ~11 | PIM (note/task/calendar create+list), Row-Bot `save_memory`/`search_memory`, `send_email`/`list_emails`, `document_search` |

## 8. Mode-routing decision (resolved before Phase 3)

**Question:** given per-mode scoping is now mandatory, how does a user request get routed to the right agent mode?

**Options evaluated:**
- **(A) Auto-detect intent** — an intent classifier (or the LLM) reads the phrasing and silently switches modes. *Risk:* this reintroduces exactly the judgment problem Phase 1/2b exposed — the 4B is unreliable at fine-grained selection. A misclassification silently routes to a mode lacking the needed tool → the capability appears "broken" with no user-visible reason. Also adds an extra LLM round-trip (latency) before any work starts.
- **(B) Explicit mode switching** — the user selects/says the mode ("research mode", "coding mode"), UI shows current mode. *Cost:* the user must know modes exist. *Benefit:* deterministic, debuggable, no misroute; trivial to implement.

**DECISION: Hybrid, explicit-first.**
1. **Explicit is the source of truth and the default UX**: the UI always shows the current mode (a selector/segmented control), and the user can switch it directly or by voice ("switch to research mode"). A request runs in whatever mode is active. This is deterministic and sidesteps the model's selection weakness entirely.
2. **Lightweight auto-*suggestion* (not auto-switch)**: a cheap keyword/rule pass (not an LLM call) may *suggest* a better mode when the active one clearly can't serve the request ("this looks like a coding task — switch to Coding?"), but it never switches silently. Confirmation stays with the user.
3. **Never** route by asking the 4B "which mode?" — that's the unreliable path we're avoiding.

Rationale: reliability over magic. Phase 1/2b demonstrated the local model's selection judgment is the weak link; making mode explicit removes it from the critical path while a rules-based hint preserves most of the convenience. Auto-detect can be revisited if/when a stronger local model makes classification dependable.

### Confirmed OpenCode server contract (verified — Phase 3 can build on this as-is, no bridge)
Investigated `packages/opencode/src/server` directly. Both Phase-3 pillars (per-mode routing, permission/approval UX) are natively supported.
- **Per-request agent (mode):** `POST /session/:sessionID/message` (sync) or **`POST /session/:sessionID/prompt_async`** (returns 204, run in background — use this) with body `PromptInput` (`session/prompt.ts:1499`), which has `agent: Schema.optional(Schema.String)` and `model`. Set `"agent":"research"|"assistant"|"coding"` per prompt. **⚠️ Verified in Phase 3: `model` is a `ModelRef` OBJECT `{providerID, modelID}` (or null), NOT a `"provider/model"` string** — sending the string 400s. Resolved off the last user message (`prompt.ts:635-644,1170`). Enumerate valid modes via **`GET /agent`** (`Agent.Info`: name/description/mode/hidden/native — filter `hidden`, `mode:"subagent"`, **and `native` (NJ-2 fix): OpenCode's built-in `build`/`plan` agents carry `native:true`; Nightjar's own modes are `native:false`, so `native!==true` shows exactly the Nightjar modes**). ⚠️ An unknown agent name does **not** 400 — it surfaces async as a `session.error` event, so validate against `GET /agent` first. ⚠️ Setting `agent` per prompt also rewrites the session's sticky agent — UI should reflect the active mode.
- **Event stream:** `GET /event` → `text/event-stream`, **instance-wide** (all bus events, filtered only by workspace) — the client must filter by `sessionID`. First frame `server.connected`, `server.heartbeat` every 10s. Key event types: `message.part.delta` (assistant text deltas), `message.part.updated` (carries **ToolPart** with a `state.status` discriminated union pending→running→completed{output}→error — this is how tool-call cards get args+result; there is no separate `tool.*` event), `permission.asked`/`permission.replied`, `question.asked`/`.replied`/`.rejected`, `session.idle`/`session.error`, `todo.updated`, `file.edited`.
- **Permission gate (the email-approval loop):** on an "ask" gate the tool publishes `permission.asked` **then blocks** (`permission/index.ts:100-106`); payload carries `{id, sessionID, permission, patterns, metadata, always, tool:{messageID,callID}}`. Answer async via **`POST /permission/:requestID/reply`** with `{reply, message?}` where `reply ∈ "once" | "always" | "reject"` (NOT allow/deny; `"always"` appends an allow-rule so future matching asks auto-approve — ideal for "always allow sends to X"; `"reject"` + `message` returns corrective feedback). `GET /permission` lists pending (lets a reconnecting UI recover missed asks). Same pattern for the `question` tool (`POST /question/:requestID/reply {answers}`).
- ⚠️ **No timeout on pending permissions** — a blocked ask waits **indefinitely** until replied (or the instance disposes). The UI must surface pending asks prominently and wire **`POST /session/:sessionID/abort`** as an escape hatch (a walked-away user otherwise wedges the run).
- **Auth:** HTTP Basic, enforced **only** when `OPENCODE_SERVER_PASSWORD` is set (default local server is open). When set, pass `Authorization: Basic …` header **or** `?auth_token=<base64(user:pass)>` query param — the query form is required for the SSE `GET /event` since browser `EventSource` can't set headers.
- **Two-stream UI:** OpenCode SSE (chat/tools/permissions/questions) + Nightjar side-channel WS (wake/transcription/orb/tts) — disjoint, multiplex both.
- **Net:** no custom event bridging or per-mode process fan-out needed. The only build-around items are (1) client-side `sessionID` filtering of the instance-wide stream and (2) the unbounded permission block → surface + abort.

**What:** give the agent the ability to *see and operate the user's desktop* — not just files/browser/email, but arbitrary native apps — via a screenshot → grounding-model → coordinates → **nut.js** (cross-platform native mouse/keyboard/screen automation) action loop. **Hazards flagged (do not silently resolve — surface each explicitly before/while building):**
1. **Hardware gate is non-optional.** OmniParser is a real vision model (GPU-hungry) that must coexist with the already-resident LLM + Kokoro/Whisper + vision model in the existing 6GB-VRAM contention picture (§7 Phase 2 hazard #3, still unresolved). 2. **Opt-in, not default-on.** Given the blast radius (native OS control), this is the one capability that must never activate without explicit, durable user consent — a settings toggle the user must find and flip, not a mode that appears alongside research/assistant/coding.
3. **New permission surface, same lesson as the Phase-3 bug.** Every nut.js action (click/type/keypress) must be gated through the OpenCode `permission` field exactly as edit/write/bash/send_email are (see the CLAUDE.md rule this phase produced) — a `tools:{x:true}` map here would be worse than the Phase-3 bug, since it would silently auto-approve native OS control.
4. **Grounding-model tiering needs its own escalation/timeout logic** (extends the wall-clock-timeout rule below) — an ambiguous screen must not cause an unbounded escalation loop between tiers.
5. **Undo is not generally possible** for OS-level actions the way `git-gate` (§7 Phase 1.5) rolls back file edits — a wrong click can have irreversible real-world effects (sent messages, deleted items in third-party apps, purchases). The safety harness for this phase needs a **pre-action confirmation/preview step** (e.g. show the intended click target highlighted before executing), not just a post-hoc approval gate.
**Status:** scoping only — no code yet.

### Odysseus fork + one-command installer
**What:** two coupled pieces of real-world-distribution hardening.

1. **Fork `pewdiepie-archdaemon/odysseus` to our own GitHub**, and point the existing `research/odysseus` **submodule at the fork** instead of upstream. Rationale: the submodule (and therefore Nightjar's whole Odysseus tier) currently has a **single point of failure** — if the upstream repo is deleted, renamed, made private, or force-pushes away the pinned commit `5acd0cea`, a fresh `git submodule update --init` breaks. A fork we control removes that dependency on someone else's repo staying available.
   - **AGPL is NOT affected by who hosts the code.** Forking does not remove or dilute any obligation: Odysseus is AGPL-3.0-or-later, the license travels with the code regardless of the host, so **all attribution + source-availability requirements still apply** to the fork exactly as they do upstream. The fork must keep Odysseus's `LICENSE`/`ACKNOWLEDGMENTS.md`/`licenses/` intact, and Nightjar's own AGPL source-availability duty (§0, §4) is unchanged. (Practically the fork makes compliance *easier*: the source we point users at is guaranteed to stay reachable.) Nightjar's two integration changes stay as the patch in `phase2-odysseus/odysseus-patches/` — the fork can stay a clean mirror, or the patch can be committed into the fork later; either way the patch/attribution story is preserved.

2. **A real one-command installer** — a single script/URL (e.g. `curl … | bash` or one `./install`) that takes a **stranger's unknown-state laptop** from nothing to a running Nightjar, not just our already-provisioned dev machines. It must handle, with detection + graceful install/guidance at each step:
   - **clone + submodule init** (fetch Odysseus from the fork);
   - **dependency detection/install** — Node, Python (+ venvs), and Ollama, checking what's already present and installing/prompting for what's missing, across the platforms we target;
   - **model download** — pull the local LLM + Kokoro/Whisper/wake models (sized/guided by llmfit's hardware fit, §7 Phase 2b);
   - **setup** — the current `scripts/setup.sh` steps (patch apply, deps, `NIGHTJAR_ROOT`), generalized for machines that are not ours.
   `scripts/setup.sh` today assumes a mostly-provisioned dev box; the installer is the productized, robust-on-a-clean-machine version. **It depends on the fork existing** (it clones/inits the submodule against the fork target), so the fork lands first.
**Status:** scoping only — no code yet.

### Cross-cutting: API key slots for cloud models (bring-your-own-key)
**What:** Nightjar is local-first and offline by default (§0 positioning), but some capabilities — a stronger cloud grounding model for Phase 5's tiered escalation, a bigger cloud model for CAD generation, or simply a user who wants a faster/better model for any mode — benefit from an optional cloud fallback. This needs to be built as **first-class, explicit settings**, not an env var or config-file hack:
- **Settings UI**: a dedicated screen (extends the Settings/health screen already scoped in §9) where a user adds a provider + API key per slot (e.g. "OpenAI key", "Anthropic key").
- **OS-keychain secure storage — not plaintext config.** Keys must be stored via the platform's native credential store (macOS Keychain / Windows Credential Manager / Linux Secret Service, e.g. via a library like `keytar`/`@napi-rs/keyring`), never written into `opencode.json`, a `.env` file, or any other plaintext file that could end up in a backup, a git commit, or a support screenshot.
- **Explicit "local vs bring-your-own-key" separation in the UI** — this is a positioning requirement, not just a technical one: Nightjar's entire pitch is offline/local-first privacy, so any screen involving a cloud key must visually and textually distinguish "runs on your machine, nothing leaves it" from "this specific action will be sent to \<provider\>" — no ambiguous blending of the two into one generic "model" picker. The user must always be able to see, at a glance, which capabilities are pure-local and which are opted into a cloud round-trip.
- **Scope of use** — a BYOK key should be usable only where the user has explicitly enabled it (e.g. "use my OpenAI key for Phase 5 grounding escalation" as a separate toggle from "use my OpenAI key for chat"), not a single global switch that silently reroutes everything through the cloud once any key is present.
**Where this sits in sequencing — ✅ DONE (shipped).** BYOK cloud-key slots shipped in **PRs #6 and #8** (this supersedes the earlier "deferred" framing): encrypted-at-rest key storage (Electron `safeStorage`, OS keychain), non-standard `NIGHTJAR_BYOK_*` env injection referenced by `opencode.json`, a model switcher + unmissable cloud indicator, a per-provider settings panel with local-vs-BYOK visual separation, and a 3-state key-storage mode (`encrypted`/`insecure`/`unavailable`) that wires the `NIGHTJAR_BYOK_ALLOW_INSECURE` test hatch through the UI honestly. **Manual verification complete.** The scoping above is the shipped design of record; OpenRouter's rate-limit auto-switch (order #2) builds directly on it. (Its original consumers — Phase 5 grounding escalation, Phase 6 CAD generation — remain optional cloud enhancements that can opt in per the scoped-use rule above.)
**Status:** ✅ **DONE** (shipped, PRs #6/#8; manually verified).
